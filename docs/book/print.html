üß™ Cleanroom Policy for JamLiquor

Overview

JamLiquor is a cleanroom Rust implementation of the JAM Protocol. It has been built from scratch to ensure:

  • JAM Prize eligibility
  • Legal and ethical separation from existing implementations
  • Trust in code provenance and conformance

This document outlines how we ensure our cleanroom standards.

‚úÖ What Cleanroom Means in JamLiquor

A cleanroom implementation ensures:

  • No direct copying of JAM source code (e.g. from ParityTech or Web3 Foundation repositories)
  • Independent reimplementation based on publicly available specifications
  • Transparent commit history, publicly timestamped
  • Documented references and architecture reasoning

üîê Cleanroom Principles

  1. No source reuse from:

    • Parity‚Äôs JAM reference clients
    • Polkadot node source code
    • Any proprietary JAM-related implementation

  2. Allowed materials:

    • JAM Graypaper (https://github.com/gavofyork/graypaper)
    • JAM Prize specs (https://hackmd.io/@polkadot/jamprize)
    • JAM-related public blog posts (e.g. kianenigma, parity forums)
    • Official documentation and diagrams (HackMD, GitHub wikis)

  3. Codebase boundaries:

    • All core components (importer, authorer, vm) are written independently
    • Any reused crates are open-source and third-party (e.g. blake3, dilithium, tinyml)
    • Modular extensions are explicitly separated from JAM core logic

  4. Contributors declare compliance:

    • All contributors agree to cleanroom principles
    • PRs involving potentially derivative logic are flagged and reviewed

üìù Documentation Trail

To support auditing and open collaboration:

  • All modules include top-level comments referencing JAM specs
  • Significant design decisions are logged in docs/decisions/
  • Any changes in architectural interpretation are documented

‚öñ License & Legal

JamLiquor is licensed under GPL-3.0, with:

  • Explicit declaration of cleanroom intent
  • No reuse of GPL-incompatible sources

We welcome external audits, JAM Prize assessments, and community reviews to validate this policy.

üì¨ Questions or Concerns?

If you believe any part of JamLiquor‚Äôs code violates cleanroom policy, please open a GitHub issue or contact us directly. Transparency is essential to our mission.

üì¶ EDGE.md ‚Äî Lightweight by Design

üîß Principle: Lightweight by Design

If your blockchain needs a server farm, it isn‚Äôt decentralized.

This raises an important question:

How can low-resource devices run a blockchain at all?

This document answers that by outlining how JAM Protocol + PolkaVM + collective edge computing solves the challenge.

‚ö†Ô∏è The Concern: Blockchains Need Power

Traditional blockchains demand:

  • üîê Heavy cryptographic operations (e.g. ECDSA, hashing)
  • üßæ Massive state and history storage (e.g. 500GB+)
  • üåê High-bandwidth syncs

Lightweight devices (e.g., 64MB RAM, 1GHz CPU) lack:

  • Enough memory for full state
  • CPU for global consensus
  • Storage for the full chain

So how can they run JamLiquor?

‚úÖ The JAM + PolkaVM Solution

1. Efficient Execution via PolkaVM

  • RISC-V based, 64-bit architecture
  • ‚â§128KB per instance
  • JIT-compiled smart contract execution
  • Native compatibility with embedded edge CPUs

2. Sharding & Role Specialization

  • JAM splits execution roles: importers, authorers, validators
  • A device only needs to fulfill one role (not all at once)
  • Importers handle state validation; Authorers handle block production

Low-resource nodes can:

  • ‚úÖ Validate local state (importer)
  • ‚úÖ Execute contracts for specific domains (e.g., IoT region)
  • ‚ùå Skip global history or full-chain validation

3. Minimal Storage & Pruned State

  • Nodes only store the latest state and headers
  • JAM supports deterministic re-execution
  • Offload historical data to archive or mid-tier nodes

Example:

  • Milk-V Duo stores 1MB snapshot
  • Contract + state = 2‚Äì5MB
  • No gigabyte chain bloat

4. Network Throughput via Collective Bandwidth

  • Devices sync only relevant data via gossip or light protocols
  • Fetch proofs or state deltas on-demand
  • Use compressed messages (~KB/sec)

10,000 devices syncing 1KB/sec = 10MB/sec aggregate bandwidth

5. Swarm-Based Computation & Redundancy

  • 1 device = 1GHz
  • 1,000 devices = 1,000GHz (distributed)
  • Total network throughput scales with adoption

Redundancy = resilience; if one fails, others take over

üåÜ Smart City Example

Use Case: 10,000 Milk-V Duo devices ($5 each)

Each:

  • Monitors air quality (sensor input)
  • Executes simple contract (if pollution > X, alert)
  • Sends alert as a JAM transaction

Network:

  • Authorers compile blocks
  • Importers verify execution
  • Mid-tier relayers sync light clients to full state

‚úÖ No datacenter. ‚úÖ No single point of failure. ‚úÖ True decentralization.

üîê PQC.md ‚Äî Quantum Resistance is Non-Negotiable

üõ°Ô∏è Principle: Quantum Resistance is Non-Negotiable

The cryptography of today won‚Äôt survive the quantum world of tomorrow.

JamLiquor integrates post-quantum cryptography (PQC) by design‚Äînot as a patch, but as a core modular extension that keeps the network resilient into the 2030s and beyond.

This document outlines how PQC works in JamLiquor, and how it can be adopted without sacrificing JAM compatibility.

‚ö†Ô∏è The Threat of Quantum Computing

Algorithms like Shor‚Äôs and Grover‚Äôs could soon break:

  • RSA, ECDSA
  • sr25519 (used in most Substrate-based chains)
  • BLS and SNARK-based crypto

Quantum-resilient systems need to:

  • Use lattice-based or hash-based cryptography
  • Prepare for migration paths today‚Äînot after compromise

‚úÖ JamLiquor's PQC Strategy

JamLiquor uses modular PQC extensions:

1. Dilithium (CRYSTALS)

  • Lattice-based digital signature
  • NIST-standardized (2022, 2024)
  • 2‚Äì4KB signature size (vs. 64B for ECDSA)
  • Suitable for JAM validators, work reports, judgments

2. Kyber (optional key exchange)

  • Module for encrypted messaging or shared secrets
  • JAM doesn‚Äôt require this, but future parachains might

3. XMSS (Hash-Based)

  • Stateless variant for resource-constrained validators
  • Very fast signing, longer verification
  • Used in certain bootstrapping tasks

üîÑ JAM Compatibility

  • PQC support lives in jamliquor-pqc/
  • Signature schemes are interchangeable via runtime contexts
  • Work Reports, validator judgments, and transaction signers can toggle PQ or classical modes
  • On-chain verifiers are optimized for batch validation

JAM‚Äôs modular signature layers make it ideal for cryptographic flexibility.

üß™ Module: jamliquor-pqc

  • Rust-first (via pqcrypto, orion, dilithium, kyber)
  • Selectable via Cargo feature flags (e.g. --features pq-dilithium)
  • Signature abstraction implements JAM-style Signer/Verifier traits

Storage Impact

  • Public key: 1‚Äì2KB
  • Signature: ~2.4‚Äì4KB

Performance:

  • Sign: ~1‚Äì2ms
  • Verify: ~2‚Äì5ms

üîê Use Cases

üîí Validator Judgments

  • Validators sign reports using Dilithium
  • Proofs are verifiable by importers

üîè Quantum-Safe Work Reports

  • Authorers can sign blocks or extrinsics with PQ keys

üóÉ Future Key Migration

  • Dual-signature support enables hybrid rollout
  • Classical + PQ keys for seamless transition

üõ† Deployment Tips

  • Use secure RNG hardware for PQ key generation
  • Limit bandwidth overhead by batching PQ signatures
  • Compress keys/signatures for gossip propagation

‚úÖ Summary

Quantum attacks are not theoretical‚Äîthey‚Äôre inevitable.

JamLiquor doesn‚Äôt wait for catastrophe. It makes PQC native, modular, and future-safe.

Security delayed is security denied.

‚ö†Ô∏è The Threat of Quantum Computing

Algorithms like Shor‚Äôs and Grover‚Äôs could soon break:

  • RSA, ECDSA
  • sr25519 (used in most Substrate-based chains)
  • BLS and SNARK-based crypto

Quantum-resilient systems need to:

  • Use lattice-based or hash-based cryptography
  • Prepare for migration paths today‚Äînot after compromise

‚úÖ JamLiquor's PQC Strategy

JamLiquor uses modular PQC extensions:

1. Dilithium (CRYSTALS)

  • Lattice-based digital signature
  • NIST-standardized (2022, 2024)
  • 2‚Äì4KB signature size (vs. 64B for ECDSA)
  • Suitable for JAM validators, work reports, judgments

2. Kyber (optional key exchange)

  • Module for encrypted messaging or shared secrets
  • JAM doesn‚Äôt require this, but future parachains might

3. XMSS (Hash-Based)

  • Stateless variant for resource-constrained validators
  • Very fast signing, longer verification
  • Used in certain bootstrapping tasks

üîÑ JAM Compatibility

  • PQC support lives in jamliquor-pqc/
  • Signature schemes are interchangeable via runtime contexts
  • Work Reports, validator judgments, and transaction signers can toggle PQ or classical modes
  • On-chain verifiers are optimized for batch validation

JAM‚Äôs modular signature layers make it ideal for cryptographic flexibility.

üß™ Module: jamliquor-pqc

  • Rust-first (via pqcrypto, orion, dilithium, kyber)
